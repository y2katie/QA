Keyboard Operability Testing

Ensuring all interactive elements are accessible via keyboard navigation (WCAG 2.1.1 & 2.1.2)

High-Level Summary

Objective: Verify that all interactive elements can be operated using only a keyboard, without creating focus traps that prevent users from navigating away.

Impact: Critical for users who cannot use a mouse due to motor disabilities, temporary injuries, or assistive technology dependencies.

Success Criteria: 2.1.1 (Keyboard) & 2.1.2 (No Keyboard Trap)

Structured Breakdown

2.1.1 Keyboard Accessibility

  • Tab Navigation: All focusable elements receive focus via Tab/Shift+Tab
  • Activation: Enter/Space keys activate buttons, links, and controls
  • Arrow Keys: Navigate within composite widgets (menus, tabs, grids)
  • Escape Key: Closes modals, dropdowns, and cancels operations
  • Custom Controls: Complex widgets implement expected keyboard patterns

2.1.2 No Keyboard Trap

  • Focus Freedom: Users can move focus away from any component
  • Modal Management: Modals trap focus but provide escape mechanisms
  • Plugin Content: Embedded content doesn't create permanent traps
  • Standard Navigation: Tab/Shift+Tab always work to move focus

Focus Management Testing

Ensuring logical focus order and preventing unexpected context changes (WCAG 2.4.3, 3.2.1, 3.2.2)

High-Level Summary

Objective: Verify focus moves predictably and that focusing or inputting data doesn't cause unexpected page changes or navigation.

Impact: Prevents disorientation and ensures users maintain control over their browsing experience.

Success Criteria: 2.4.3 (Focus Order), 3.2.1 (On Focus), 3.2.2 (On Input)

Structured Breakdown

2.4.3 Focus Order

  • Logical Sequence: Focus follows reading order (left-to-right, top-to-bottom)
  • Meaningful Order: Focus sequence preserves meaning and operability
  • Skip Links: Provide shortcuts to main content areas
  • Modal Focus: Focus moves to modal when opened, returns when closed

3.2.1 On Focus & 3.2.2 On Input

  • No Auto-Submit: Forms don't submit when fields receive focus
  • No Auto-Navigation: Focus doesn't trigger page changes
  • Predictable Changes: Any changes are clearly indicated beforehand
  • User Control: Users initiate all context changes explicitly

Pointer & Click Behavior Testing

Testing pointer cancellation and interaction patterns (WCAG 2.5.2)

High-Level Summary

Objective: Ensure pointer interactions can be cancelled and don't trigger unintended actions.

Impact: Prevents accidental activations for users with motor disabilities or tremors.

Success Criteria: 2.5.2 (Pointer Cancellation)

Structured Breakdown

2.5.2 Pointer Cancellation

  • Down Event Safety: No action triggered on pointer down event alone
  • Up Event Completion: Actions complete on pointer up event
  • Abort Mechanism: Can cancel by moving pointer away before release
  • Undo Functionality: Actions can be reversed if immediately triggered

Labels & Names Testing

Testing label matching and accessible name/role/value implementation (WCAG 2.5.3, 4.1.2)

High-Level Summary

Objective: Ensure visible labels match programmatic names and all components expose proper accessibility information.

Impact: Critical for speech input users and assistive technology compatibility.

Success Criteria: 2.5.3 (Label in Name), 4.1.2 (Name, Role, Value)

Structured Breakdown

2.5.3 Label in Name

  • Visible Text Match: Accessible name contains visible label text
  • Speech Input: Users can say visible text to activate controls
  • Icon Labels: Icon-only buttons have appropriate accessible names
  • Consistent Naming: Same function = same name across the site

4.1.2 Name, Role, Value

  • Name Definition: Purpose/function is programmatically determinable
  • Role Definition: Purpose/function is programmatically determinable
  • State/Value: Current state and values are exposed to AT
  • Change Notification: State changes are announced to AT

Visual Feedback Testing

Testing hover and focus content behavior for consistency and accessibility (WCAG 1.4.13)

High-Level Summary

Objective: Ensure content that appears on hover or focus is dismissible, hoverable, and persistent enough for interaction.

Impact: Critical for users with motor disabilities, low vision, or those using assistive technology.

Success Criteria: 1.4.13 (Content on Hover or Focus)

Structured Breakdown

1.4.13 Content on Hover or Focus Requirements

  • Dismissible: Can be dismissed without moving hover or focus (usually Escape key)
  • Hoverable: If triggered by hover, the additional content can be hovered over
  • Persistent: Content remains visible until hover/focus is removed or user dismisses it
  • No Interference: Additional content doesn't obscure other page content
