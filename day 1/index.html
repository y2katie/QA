The Foundation of Screen Reader Accessibility

Every interactive element on a web page must have three key properties for screen readers to properly announce them:

Name

What the element is called

Role

What type of element it is

Value

What state or content it has

Fundamentals: The Three Properties

Understanding Name, Role & Value

Screen readers rely on these three properties to create meaningful announcements for users. Let's explore each one:

Interactive Demo: Listen to the Difference

Property Breakdown

Name: What is this element called?

The accessible name is how screen readers identify an element to users.

Sources of Accessible Names (in priority order):
  1. aria-labelledby - References another element's text
  2. aria-label - Direct text label
  3. Label element - Associated form label
  4. Element content - Text inside the element
  5. title attribute - Tooltip text (last resort)
Good Name
<button aria-label="Close dialog">×</button>

Screen Reader: "Close dialog, button"

Poor Name
<button>×</button>

Screen Reader: "Times, button" (confusing!)

Role: What type of element is this?

The role tells screen readers what kind of interaction or content to expect.

How Roles are Determined:
  1. Explicit ARIA role - role="button"
  2. Implicit HTML semantics - <button> has implicit role="button"
  3. Default generic role - <div> has no semantic meaning
Button Role
<button>Submit</button>
<div role="button">Submit</div>

Both announce as "Submit, button"

Link Role
<a href="/home">Home</a>
<div role="link">Home</div>

Both announce as "Home, link"

Value: What is the current state or content?

The value represents the current state, content, or properties of an element.

Types of Values:
Form Values

Current input content

<input type="text" value="John Doe">

"John Doe" (current value)

State Values

Current element state

<button aria-pressed="true">Bold</button>

"Bold, toggle button, pressed"

Property Values

Additional properties

<input aria-required="true">

"required" (additional property)

Advanced QA Scenarios

What Would Happen If...

A form field has both a label and aria-label with different text? Which one would the screen reader announce and why?

Answer: aria-label takes precedence over the label element. The screen reader would announce the aria-label text. This could confuse users who can see the visual label but hear different text.

QA Test: Always verify that aria-label matches or enhances the visible label text.

Real-World Debugging

Users report that a "Save" button sometimes announces as "Save" and sometimes as "Saving..." but the visual text never changes. What's likely happening?

Likely Cause: JavaScript is dynamically updating the aria-label during the save process.

QA Tests:

  • Check if aria-label changes during interaction
  • Verify the button is properly disabled during save
  • Test with screen reader to confirm announcements
  • Ensure visual feedback matches audio feedback
